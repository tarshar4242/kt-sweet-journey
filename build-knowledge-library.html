我沒有學過寫網頁。但我有一個問題：我讀的論文放在哪裡，指導教授怎麼看到我有多認真？我打開 AI，說了一句話，這個網站就開始長出來了。

STEP 01

起點：先有需求，才有工具

定義你要解決什麼問題，不是先問「該學哪個工具」。

發生什麼事

一開始我沒有想「我要學做網頁」。我想的是一個很具體的困擾：我讀的論文散在電腦各處，指導教授要看我讀了什麼、讀到哪裡，我只能一篇一篇傳檔案。我把這個困擾拆成四個條件，寫給 AI 看：

  • 論文 PDF 要能線上看
  • 要能搜尋關鍵字
  • 有人分享連結就能直接打開
  • 不用花錢，也不用管伺服器

AI 給的答案是 GitHub Pages：一個免費、不用自己架主機、丟檔案上去就能變成網址的服務。

我學到什麼

先想清楚你要什麼，AI 才能給你對的路。
需求排在工具前面，順序不能倒

可複製的做法

🍀 打開 AI 之前，先花五分鐘寫下你的「我要解決什麼問題」清單，越具體越好（像「要能線上看」「不用花錢」這種可以打勾的條件），再把清單丟給 AI 問「這樣的話，你會建議我用什麼方式做？」
STEP 02

架構：資料夾就是你的設計圖

用結構思考，不用程式思考。

發生什麼事

我不會寫程式，但我會整理資料夾。所以我用自己最熟的方式先把東西分類：一篇論文，一個資料夾，資料夾裡放它的原文和一份說明。AI 照著這個邏輯，把它變成一個網站的骨架：

thesis-library/ ├── index.html ← 首頁總覽 ├── A04/ ← 每篇論文一個資料夾 │ ├── README.md │ └── A04原文.pdf ├── A05/ ...

每篇論文都住在自己的資料夾裡。結構清楚，AI 就不會搞混，我也不會搞混。

我學到什麼

用你能理解的方式組織資料，AI 幫你變成網頁。
你不需要懂資料庫，你只需要懂怎麼分類

可複製的做法

🍀 先用你原本就會的方式（資料夾、Excel 表、清單都可以）把東西分類好，再請 AI「照這個結構幫我變成網頁」。你負責邏輯，AI 負責轉換。
STEP 03

視覺設計：我說「淺綠文青風」，它真的懂

用形容詞對話 AI，不用學設計術語。

發生什麼事

我完全不懂配色理論、不懂什麼是留白比例。我只跟 AI 說了一句話：

「淺綠、文青、像獨立書店，不要太科技感。」

AI 把這句話翻成具體的設計語言：

  • 背景色 #f4f0e8：溫暖奶白
  • 主色 #8cbd7d：薄荷鼠尾草綠
  • 字型 PingFang TC：繁中乾淨易讀
  • 米色卡片、細邊框、輕陰影

我學到什麼

你不需要學設計，你只需要描述你的感覺。
形容詞是你跟 AI 之間最快的共同語言

可複製的做法

🍀 想不出配色名詞沒關係，找一個你喜歡的地方或東西當比喻（獨立書店、日式雜貨、老宅咖啡廳），直接告訴 AI「像那種感覺」，它會幫你翻成顏色和字型。
STEP 04

踩坑：白色區塊是怎麼來的？

遇到 Bug，讓 AI 解釋給你聽，不要只求「修好就好」。

發生什麼事

第一版網站上線後，我發現每張論文卡片右邊都多了一個莫名其妙的白色方塊，怎麼看都不對勁。我把畫面截圖給 AI，問「這是什麼？為什麼會這樣？」

AI 解釋：HTML 規定 <a> 裡面不能再放 <a>。外層卡片和下載按鈕都是連結，瀏覽器會自動截斷，白塊就跑出來。
修法：外層改 div + onclick，按鈕維持 a。

我學到什麼

Bug 不可怕。理解根本原因，才是真的學到。
問「為什麼」比問「怎麼修」更值得

可複製的做法

🍀 遇到畫面壞掉，先截圖丟給 AI，多問一句「為什麼會這樣」。你不用記住原理，但聽過一次原因，下次遇到類似狀況你會更快抓到方向。
STEP 05

搜尋功能：15 行程式碼

功能不用複雜，夠用就是好設計。

發生什麼事

我以為「搜尋功能」一定要接資料庫、要很複雜的技術。結果 AI 給我的答案只有短短幾行：

function filterCards() { const q = searchInput.value.toLowerCase(); cards.forEach(card => { const match = card.textContent.includes(q); card.style.display = match ? '' : 'none'; }); }

輸入關鍵字，卡片即時篩選。不用後端、不用資料庫、不用 API。

我學到什麼

能解決 80% 的需求，就不需要 100% 的複雜度。
先問「夠用嗎」，不要先問「夠厲害嗎」

可複製的做法

🍀 提需求時先講「我要解決的是什麼」，不要先預設要做到多厲害。跟 AI 說「我只是想要能快速找到那篇」，它給你的方案通常比你想像的簡單很多。
STEP 06

部署：從本機到全世界，只要三行指令

GitHub Pages 部署全流程。

發生什麼事

網頁做好放在自己的電腦裡，沒有人看得到。要讓它變成一個誰都能點開的連結，我打的指令其實只有三行：

git add . git commit -m "更新論文庫" git push origin main

打完這三行，兩分鐘後全世界都看得到。網址像這樣：tarshar4242.github.io/thesis-library/

我學到什麼

不用租伺服器。不用買網域。不用設定 HTTPS。
「上線」這件事，門檻比你想的低很多

可複製的做法

🍀 這三行指令請 AI 一步一步教你打一次，打過一次你就會了。之後每次更新內容，重複同樣三行就好，不需要每次都重新學。
STEP 07

工作流程：我是怎麼跟 AI 協作的

把 AI 當作一個不會累的設計師。

發生什麼事

整個過程我沒有一次把需求一口氣講完，而是來回四個動作，一直重複：

1
描述目標

「我要在每張卡片底部加檔案按鈕」，講清楚我要什麼，不用懂術語

2
確認方向

AI 提方案，我說好或修正，我永遠是拍板的人

3
遇到問題就截圖

「為什麼長這樣？」，用畫面溝通，比用文字描述準

4
理解再繼續

每個 Bug 都要求它解釋原因，不是修好就跳過

我學到什麼

不需要每行程式碼都看懂，但每個決定都要知道為什麼。
你不是外包給 AI，你是在跟它一起做決定

可複製的做法

🍀 每次跟 AI 對話都用這四步循環：說目標 → 看方案 → 有問題就截圖問 → 聽懂原因再繼續。你不用變成工程師，但你要一直知道「現在在做什麼、為什麼這樣做」。

給你的三步驟行動建議：明天就能開始

這個網站記錄的不只是 8 篇論文。它記錄的是一個人在 2026 年，選擇用 AI 作為工具，把「我有在認真做研究」變成任何人都能打開的連結。你也可以，從明天開始。

1

寫下你的「我要解決什麼問題」清單

不用想工具、不用想技術，先寫三到四條你真正卡住的地方（像起點那一章的四點）。清單越具體，AI 給的方向越準。

2

把你手上的資料先分類

用資料夾、用 Excel、用清單都可以，先照你自己看得懂的邏輯整理好，再交給 AI「照這個結構幫我變成網頁」。

3

用一句話形容你想要的感覺，打開 AI 開始問

不用學設計、不用學程式，找一個你喜歡的比喻（一間店、一個氛圍），描述給 AI 聽，然後照著這篇的七個章節，一步一步走一次。

你不需要是工程師。你需要的只是一個問題，然後開始問。
