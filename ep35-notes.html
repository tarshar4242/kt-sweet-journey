一句話先懂Harness（馬具）＝包在大模型外層、用來駕馭它的整套軟體系統。公式：完整 Agent − 核心大模型 ＝ Harness。它不是讓模型更聰明，而是讓模型在你的系統裡穩定做事。
從玩具到資產Harness 的價值，是把 AI 從「測試很強、上線就掉鏈」的玩具，升級成可控、可觀測、可維護的工業資產。

CHAPTER 01

1. 為什麼需要 Harness：大模型的先天缺陷

為何更換更強的模型，也救不了失敗的工作流？

大模型本質上是無狀態的（Stateless）——就像一個擁有博士學位、卻每隔幾分鐘就會失憶的學者。當你把它放進需要連續執行三十分鐘、一小時的工作流程，會發生「三件慘事」：

失敗模式發生什麼
上下文腐爛模型忘記最初的指令，越做越偏。
工具調用幻覺參數傳錯，陷入死循環。
狀態丟失任務做到一半就斷線、過早停止。
🐴 這就是為什麼很多人在小範圍測試覺得 AI 超強，一推到真實生產線，成功率連 70% 都不到。換更強的模型、調更精的提示詞，都救不了這個結構性問題——所以我們需要全新的工程思維。
CHAPTER 02

2. AI 工程的三次遷徙

從『聽懂』→『拿到資訊』→『持續做對』。

圖解｜AI 工程三次重心遷移。前兩階段仍然有用，但無法保證第三件事——在長鏈路任務中穩定、可靠地把活做完。
階段解決什麼侷限
① 提示詞工程意圖表達——把話說清楚（角色、約束、範例）。再精緻也無法彌補模型缺失的知識，更無法管理長鏈路狀態。
② 上下文工程／RAG在對的時機把正確資訊、工具、歷史放進上下文窗口。窗口容量有限，壓縮就丟資訊，模型反覆犯同樣的錯。
③ Harness Engineering在真實世界長時間、低容錯地持續做對。需要「監督與糾偏」機制——這正是前兩者缺的。
CHAPTER 03

3. Harness 的定義：Agent − Model = Harness

Harness 原意是馬具，象徵對力量的約束與控制。

大模型就像算力強大卻容易跑偏的野馬；Harness 就是包覆模型外層、用來控制與駕馭它的軟體作業系統。業界有一個乾淨的定義公式：

完整 Agent － 核心大模型 ＝ Harness
記憶管理、工具代碼、權限控管、流程編排、錯誤處理、評估機制……扣掉中間那顆模型，剩下全部都是 Harness。
🎯 視角轉換：Harness Engineering 不研究「怎麼讓模型更聰明」，而是研究「怎麼讓模型在我們的系統裡安穩工作」。
CHAPTER 04

4. 成熟 Harness 的六層架構（本集核心）

一個成熟的 Harness 可以拆成六層，由內而外把野馬套牢。

1

上下文管理與角色邊界

模型不該一開始就被塞入所有規範。用漸進式載入（Progressive Loading）：核心配置只保留最關鍵的規則與索引，執行到特定步驟才動態讀取外部細則——保持上下文乾淨。

2

外部工具與動態技能系統

沒有工具，模型只是文字預測器。關鍵不只是把 API 丟給它，而是解決給什麼、何時用、結果如何反饋；用 MCP 等協定接資料庫，把操作邏輯封裝成 Skills，並清洗工具回傳的雜訊。

3

任務編排與執行邏輯

建立標準執行軌道，而非讓模型想到哪做到哪。拆成理解目標→判斷資訊→調用工具→分析結果→驗證輸出，採 ReAct 或 Plan-and-Execute；進階可用多個 Sub-agents 協作（子 Agent 上下文互不干擾）。

4

記憶系統與狀態持久化

把失憶的大腦變成會累積經驗的打工仔。管理當前任務狀態、會話中間結果、長期記憶與偏好；狀態存到外部儲存，任務中斷還能從斷點精準恢復。

5

質量評估與觀測機制

模型自評一定「王婆賣瓜」。要把生成（Generator）與評估（Evaluator）拆成兩個獨立實體，用日誌、指標、模擬操作檢驗，把 AI 推理轉成成功率、耗時、魯棒性等可量化指標。

6

約束、錯誤攔截與自動恢復

「該錯的事一定會錯。」設輸入過濾（擋 Prompt 注入）與輸出校驗；出錯時把錯誤打包回傳給模型引導它反思重生成；反覆犯的同類錯，直接在代碼層寫死規則堵住。

圖解｜六層 Harness 把「無狀態的野馬」層層包覆，從上下文邊界一路到錯誤防護網。
CHAPTER 05

5. OpenAI／Anthropic 的真實做法

六層架構不是理論——大廠早已這樣落地。

  • Human Steer, Agents Execute（OpenAI）：人類掌舵、代理執行——刻意保留人類的中繼觀察點，而不是全自動放生。
  • 生成與評估分離（Anthropic）：Planner／Generator／Evaluator「三權分立」，避免模型自評的盲點；評估不過就「打回重構」。
  • 代碼層硬規則：對於模型反覆犯的同類錯誤，工程師直接在代碼寫死規則（例：強制置換錯誤背景色）從根本堵死。
⚠️ 關鍵心法：「代碼主導流程，模型負責單點突破」——把不可控的大腦，放進一個有結構、有防線的工作環境裡。
實 作

6. 實作：每日 AI 新鮮事產生器

不寫程式、不用 API，只用 Gemini Gem＋NotebookLM＋複製貼上，做出一個具備六層 Harness 結構的小工具。

平台分工（為什麼分兩個平台？）

平台角色
NotebookLM長期記憶＋來源權威：訂閱 15+ 個 AI 新聞來源、儲存歷史稿件避免重複。
Gemini Gem執行引擎＋任務編排：每天拉資料、寫稿、配圖、輸出。
Nano BananaGemini 2.5 Flash Image，負責每日封面圖。
Google Docs最終輸出與歷史歸檔（可下載 docx）。
🔑 「生成」與「記憶」、「資料」與「執行」刻意分離，中間靠你的「複製貼上」接力——這正是 Human Steer, Agents Execute 的體現。

六層怎麼落地（對照上面的架構）

  1. 第一層｜角色邊界：Gem 系統指示只放「角色定位＋核心邊界＋模組索引」，用指令關鍵字（【啟動】【素材】【寫稿】【配圖】【評估】【輸出】【歸檔】）漸進式載入。
  2. 第二層｜工具與技能：NotebookLM 當「人類仲介的工具」，寫稿／配圖／輸出各封裝成獨立模組 Prompt。
  3. 第三層｜任務編排：把每天拆成 7 個關卡、一次一關，每關用 ReAct 風格輸出〔思考〕〔行動〕〔預期〕〔觀察〕。
  4. 第四層｜記憶：當日狀態放 Gem、近期主題放 Google Docs 工作日誌、長期素材放 NotebookLM；開工前先「載入過去 7 天主題」避免重複。
  5. 第五層｜評估：另開一個「主編評估官」Gem，用七大維度逐項打分（滿分 35、門檻 30），不過就退稿重寫。
  6. 第六層｜防護網：輸入過濾（擋注入與越界）＋輸出校驗（6 項硬規格），不過自動 Retry 最多 2 次。

三個關鍵 Prompt（手冊原文）

第一層 · Gem 系統指示（漸進式載入骨架）# 角色定位：你是「每日 AI 新鮮事產生器」，為台灣中文讀者整理 AI 業界動態的編輯助理… # 漸進式載入（關鍵）：依使用者輸入的「指令關鍵字」逐步進入對應模組—— ・收到「【啟動】」→ 回「已就緒，請貼入今日素材」並等待 ・收到「【素材】」+內容 → 進入分類模組 ・收到「【寫稿】」→ 進入寫稿模組 … 未收到關鍵字時，不主動執行下一步。 # 不變約束：不得編造新聞/人名/數據；任何事實必附來源；單一來源引用不超過 25 中文字。
第三層 · 主編排指令（流程鎖＋ReAct）【編排規則】每天任務分 7 關，依序通過不得跳關：①啟動→②接收素材→③分類→④寫稿→⑤自我評估→⑥配圖→⑦輸出。 每關開始前先輸出三行：〔思考〕我要做什麼、為什麼；〔行動〕即將執行的步驟；〔預期〕應產出什麼。 每關結束輸出一行〔觀察〕產出與預期是否一致。完成後只回「✅ 第 N 關完成，等待下一步」，然後停。
第五層 · 主編評估官（生成/評估分離）# 角色：嚴格的中文編輯主編，職責是挑毛病，不替作者著想。 # 七大維度逐項打分(1–5)：①事實準確度 ②重複度 ③結構完整 ④字數合規 ⑤風格一致 ⑥引用合規 ⑦台灣相關性。 # 輸出：逐項評分+總分(滿分35)+通過門檻30；未過門檻給「修訂建議清單」；發現編造事實直接判「整篇退稿」。
✅ 手冊完整收錄六層全部 Prompt（1A／2A–2D／3A／4A–4B／5A–5B／6A–6B）、每日 SOP 與故障排除 FAQ，可直接複製貼上使用。

7. 每日操作 SOP（約 15 分鐘）

熟練後一輪約 12–18 分鐘，這是每日標準流程。

步驟在哪做做什麼
① 蒐集NotebookLM查「過去 24 小時最重要的 5–8 則 AI 動態」
② 過濾NotebookLM人眼確認來源權威、日期、無重複
③ 狀態載入Gem貼上過去 7 天主題（Prompt 4A）避免重複
④ 啟動＋分類Gem【啟動】→【素材】+內容
⑤ 寫稿Gem【寫稿】（Prompt 2B）產出 800–1200 字稿件
⑥ 評估主編評估官 Gem貼稿件評分，未過門檻回原 Gem 修訂
⑦ 配圖＋輸出Gem【配圖】選 A/B → 【輸出】Markdown → 貼 Docs 下載 docx
⑧ 歸檔Gem＋Docs【歸檔】更新工作日誌主題追蹤表
「換主題不換架構——這就是 Harness 思維的可重用性。」
同一套六層架構，換掉 NotebookLM 來源就能做每日國防新聞、每週政策摘要、文獻週報。

8. 重點整理（帶走這幾句）

  • 大模型是無狀態的野馬：上下文腐爛、工具幻覺、狀態丟失——換更強的模型也救不了。
  • 三次遷徙：聽懂（Prompt）→ 拿到資訊（Context/RAG）→ 持續做對（Harness）。
  • Harness ＝ Agent − Model：記憶、工具、編排、評估、防護，全部都是 Harness。
  • 六層架構：①角色邊界 ②工具技能 ③任務編排 ④記憶狀態 ⑤評估觀測 ⑥錯誤防護。
  • 漸進式載入：核心只放索引，細節按需載入，保持上下文乾淨。
  • 生成／評估分離：不要讓模型自評（王婆賣瓜），另設獨立評估角色。
  • 狀態存外部：不靠模型記憶，存到 Google Docs／NotebookLM，可從斷點恢復。
  • 不寫程式也能做 Harness：Gem＋NotebookLM＋複製貼上，就是一個六層小工具。
「從 Prompt 到 Harness，從個人玩家到系統設計者。」
Harness 不是新技術，而是把既有工具「組裝得更可控」的工程思維。
